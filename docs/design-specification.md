## 1. プロジェクト概要

### 1.1 目的

本プロジェクトは、Joseph Campbell の「ヒーローズ・ジャーニー（英雄の旅）」理論に基づき、作家の個人的なナラティブ（自己物語）をAIで分析・変換し、12段階の物語構造に沿った完全なフィクション作品を自動生成するシステムである。

### 1.2 基本コンセプト

- **入力:** 作家自身の内面（願望・欠落・葛藤・記憶など）を構造化した自己ナラティブ
- **処理:** 複数のAIモデルによる分析→要素生成→キャラクター創造→プロット構築→物語執筆
- **出力:** 10章構成の完全な小説原稿、キャラクタープロフィール、世界観設定、ビジュアルプロンプト

### 1.3 設計思想

1. **作家の深層心理を物語に昇華する**: 入力された自己ナラティブからAIが願望・抑圧・葛藤を抽出し、キャラクターと物語に投影する
2. **ランダム性による多様性**: 生成要素の組み合わせにランダム選択を導入し、同一入力から最大100パターンの異なる物語を生成可能にする
3. **マルチモデル活用**: 分析・構造化にはOpenAI（o3-mini）、文学的執筆にはClaude、推論にはDeepSeekと、各モデルの得意領域を活かす

## 5. 入力仕様

### 5.1 作家ナラティブ入力フィールド一覧

作家が記入する13の自己ナラティブ項目。各項目は自由記述テキスト。

| 変数名 | 入力プロンプト（質問文） | 用途 |
|---|---|---|
| `author` | あなたはどんな人物ですか？思いつくままに記してください | 作家の人物像 |
| `missing` | 「私には○○が欠けている。それは自分にとって○○を象徴するものである」の形式で | 欠落しているもの |
| `status` | 成功している / かつては成功していた / 未だ成功したことがない / 成功とは何かわからない | 現在の状況評価 |
| `memories` | 現在の状況に最も強い影響を与えた過去のできごと | 重要な過去体験 |
| `mission` | 「欠けているもの」を手に入れるためにクリアすべき具体的なミッション | 課題・クエスト |
| `success` | 「欠けているもの」がいつか手に入るとイメージできるか？ | 成功の見込み |
| `loss` | 「欠けているもの」を手に入れる過程で手放さなければいけないもの | 代償 |
| `taboo` | 「欠けているもの」を手に入れようとする過程で決して破ってはいけないタブー | 制約条件 |
| `inhibit` | 「欠けているもの」を手に入れようとするときにいつも邪魔するもの | 阻害要因 |
| `daily` | 日常生活に関するキーワード | 日常世界の素材 |
| `change` | 日常生活に変化をもたらす存在 | 変化の触媒 |
| `acceptance` | 大切なものを脅かす存在と和解・許容できるか？ | 敵対者との関係性 |
| `desire` | 誰にも話すことができない秘めた願望 | 深層の欲求 |

## 6. コア関数定義

### 6.1 `ai(prompt, model="o3-mini", reasoning_effort="medium")` — テキスト生成

OpenAI ChatCompletion APIを使用してテキストを生成する。

```
入力:  prompt (str)           — AIへの指示文
       model (str)            — 使用モデル（デフォルト: "o3-mini"）
       reasoning_effort (str) — 推論レベル（デフォルト: "medium"）
出力:  str — 生成テキスト
```

- `reasoning_effort` は `"low"`, `"medium"`, `"high"` から選択可能
- タスク種別に応じて `REASONING_CONFIG` 辞書から適切な値を渡す（後述 6.6 参照）
- エラー発生時は `"エラーが発生しました: {エラー内容}"` を返す

### 6.2 `ai_list(prompt, model="o3-mini", reasoning_effort="medium")` — JSON構造化データ生成

OpenAI ChatCompletion APIを使用してJSON形式の構造化データを生成する。

```
入力:  prompt (str)           — AIへの指示文
       model (str)            — 使用モデル（デフォルト: "o3-mini"）
       reasoning_effort (str) — 推論レベル（デフォルト: "medium"）
出力:  dict — JSON形式のデータ（辞書型）
設定:  response_format = {"type": "json_object"}
       system message = "常にJSON形式で応答してください"
       user suffix = "\nJSON形式で答えろ"
```

### 6.3 `claude(prompt)` — 文学的テキスト生成

Anthropic Claude APIを使用して文学的品質の高いテキストを生成する。

```
入力:  prompt (str) — 執筆指示文
出力:  str — 生成された物語テキスト
設定:  model = "claude-3-5-sonnet-20241022"
       max_tokens = 4000
       temperature = 1（最大の創造性）
       system = "あなたは現代を代表する小説家です。"
```

### 6.4 `dsr1(prompt, model="deepseek-reasoner")` — DeepSeek推論生成

DeepSeek APIを使用してテキストを生成する（代替モデル）。

```
入力:  prompt (str) — AIへの指示文
       model (str)  — 使用モデル（デフォルト: "deepseek-reasoner"）
出力:  str — 生成テキスト
設定:  api_base = "https://api.deepseek.com"
```

### 6.5 タスク別 reasoning_effort 設定（`REASONING_CONFIG`）

タスクの複雑さに応じて `reasoning_effort` を最適化するための設定辞書。ノートブックの Cell 3 先頭で定義される。

```python
REASONING_CONFIG = {
    "analysis":        "high",    # ナラティブ分析（願望/抑圧/葛藤）
    "narrative":       "high",    # ナラティブ要素抽出
    "element_pool":    "low",     # 要素プール生成（wants/abilities/roles）
    "plot_classify":   "low",     # プロット形式分類
    "character":       "medium",  # キャラクター生成
    "plot":            "medium",  # プロット生成
    "visual":          "low",     # ビジュアルプロンプト
    "world":           "medium",  # 世界観生成
    "plot_skeleton":   "medium",  # プロット骨子(A-E)
    "integrated_plot": "high",    # 統合プロット
    "default":         "medium",  # デフォルト
}
```

**適用方針:**

| reasoning_effort | 適用タスク | 理由 |
|---|---|---|
| `"high"` | ナラティブ分析、ナラティブ要素抽出、統合プロット | 深い分析・複雑な構造把握が必要 |
| `"medium"` | キャラクター生成、プロット生成、世界観生成、プロット骨子 | バランスの取れた品質が必要 |
| `"low"` | 要素プール(100個)、プロット形式分類、ビジュアルプロンプト | 大量生成・単純変換タスク |

### 6.6 ユーティリティ関数

| 関数 | 入力 | 出力 | 説明 |
|---|---|---|---|
| `show(data)` | `dict` / `list` | 表示 | データをMarkdown形式に変換して表示 |
| `rich_print(m)` | `str` | 表示 | Markdownテキストをレンダリング表示 |

## 9. キャラクター生成システム

### 9.1 キャラクター要素プール生成

ナラティブ入力（`input`変数）から、キャラクターの構成要素を大量に生成する。

| 要素プール | プロンプト概要 | 生成数 | JSON キー |
|---|---|---|---|
| 願望リスト | キャラクターが心の中に持っている願望 | 100個 | `wants` |
| 能力リスト | キャラクターが秘めている特有の能力 | 100個 | `abilities` |
| 課題リスト | キャラクターが抱えている課題 | 100個 | `roles` |

### 9.2 キャラクターアーキタイプ定義

Joseph Campbellの元型理論に基づく4つのキャラクタータイプ。

#### 主人公（Protagonist）

```
定義:  物語の中心人物であり、冒険と試練を通じて成長し、変化する存在。
       内的および外的な葛藤を克服することで、自己実現や世界への貢献を果たす。
出力:  名前 + 400文字程度のプロフィール紹介文
入力要素:
  - 物語の構造（plot_listからランダム選択）
  - 抑圧されている自己像（narrativeからランダム選択）
  - 内面の願望（wantsからランダム選択）
  - 秘めた能力（abilitiesからランダム選択）
  - 個人的な課題（rolesからランダム選択）
```

#### 使者（Messenger / Herald）

```
定義:  主人公に知恵や助言を与え、冒険への一歩を踏み出すきっかけを作る存在。
       成長のための方向性や目的意識を提供する。
出力:  名前 + 200文字程度のプロフィール紹介文
入力要素:
  - 主人公の人物像（prompt_protagonist）
  - 物語の構造、抑圧された自己像、内面の願望、秘めた能力、個人的な課題
    （各要素プールからランダム選択）
```

#### 援助者（Supporter / Ally）

```
定義:  主人公をサポートしたり、試練を通じて成長を促す存在。
       時に助け、時に混乱をもたらしながら、主人公に新たな視点や気づきを与える。
出力:  名前 + 200文字程度のプロフィール紹介文
入力要素:  使者と同一構造（主人公の人物像 + ランダム選択された各要素）
```

#### 敵対者（Adversary / Shadow）

```
定義:  主人公が克服すべき外的・内的な障害や敵対者。
       英雄の成長を試す存在であり、恐怖、誘惑、葛藤を象徴する。
出力:  名前 + 200文字程度のプロフィール紹介文
入力要素:  使者と同一構造（主人公の人物像 + ランダム選択された各要素）
```

### 9.3 キャラクター生成ループ

```
for i in range(100):
    1. plot_listからプロット形式をランダム選択
    2. narrative, wants, abilities, rolesから各1要素をランダム選択
    3. 主人公を生成（400字プロフィール）
    4. 要素プールから再ランダム選択
    5. 使者を生成（200字プロフィール、主人公情報を参照）
    6. 要素プールから再ランダム選択
    7. 援助者を生成（200字プロフィール、主人公情報を参照）
    8. 要素プールから再ランダム選択
    9. 敵対者を生成（200字プロフィール、主人公情報を参照）
   10. 4人のキャラクターを用いて10章構成のプロットを生成
```

**設計上のポイント:**
- 各キャラクターは毎回異なるランダム要素の組み合わせで生成される
- 使者・援助者・敵対者は主人公のプロフィールを文脈として受け取る（一貫性の担保）
- 100回のループにより、同一の作家ナラティブから最大100種の異なるキャラクターセット＋プロットが生成される

## 11. ヒーローズ・ジャーニー物語構造定義

Joseph Campbellの「千の顔をもつ英雄」およびChristopher Voglerの「ライターズ・ジャーニー」に基づく11段階の物語構造。ノートブック内の `plot_structure` 変数で定義される。

### 11.1 構造一覧

| # | 段階名 | 幕 | 目的 |
|---|---|---|---|
| 1 | 日常世界 | 第一幕 | 主人公の普段の生活、環境、価値観、現状維持の姿勢を描く |
| 2 | 冒険への呼びかけ | 第一幕 | 現状から抜け出し、冒険に向かうきっかけを提示 |
| 3 | 拒否 | 第一幕 | 冒険への参加をためらうことで、冒険の重大さを強調 |
| 4 | 師との出会い | 第一幕 | 指導者との出会いで、勇気や知恵を得る |
| 5 | 第一関門の突破 | 第二幕 | 未知の世界に足を踏み入れる決定的な行動 |
| 6 | 試練、仲間、敵 | 第二幕 | さまざまな試練に直面し、仲間や敵との関係を築く |
| 7 | 最大の試練 | 第二幕 | 命をかけた試練で極限の経験をする |
| 8 | 報酬 | 第二幕 | 最大の試練を乗り越えた結果としての成果や気づき |
| 9 | 帰路 | 第三幕 | 得たものを持って元の世界へ帰る決意と過程 |
| 10 | 復活 | 第三幕 | 最後の試練で真の変化と成長を遂げる |
| 11 | 宝を持ち帰る | 第三幕 | 冒険で得た「宝」を日常世界で活かす |

### 11.2 各段階の詳細ポイント

**1. 日常世界**
- 主人公の欠点や不満、未解決の課題を提示する
- 読者が共感できる要素を強調する
- 主人公の「変化前」の基準点を明確にする

**2. 冒険への呼びかけ**
- 異常事態、事件、人物との出会いなどで現状を揺るがす
- 主人公に「選択」を迫る状況を作る

**3. 拒否**
- 恐れ、不安、責任感、無力感などの内的・外的葛藤を描く
- 拒否する理由が納得できるものにすることでリアリティを持たせる

**4. 師との出会い**
- メンターは知識やスキルだけでなく、精神的な支えとなる存在
- メンターの言葉や行動が主人公の内面に変化をもたらす

**5. 第一関門の突破**
- 「戻れない一線」を越える瞬間を強調する
- 新たな環境、ルール、危険の紹介

**6. 試練、仲間、敵**
- 友情や信頼、裏切りといった人間関係のドラマを展開
- 小さな成功と失敗を繰り返し、主人公の成長を段階的に描写

**7. 最大の試練**
- 肉体的・精神的に最も厳しい挑戦
- 主人公が「死と再生」の象徴的な体験を通じて変化する

**8. 報酬**
- 物理的な宝、知恵、自己理解、仲間との絆など多様な報酬
- 達成感と同時に、新たな課題や責任が見えてくることもある

**9. 帰路**
- 帰還途中にも新たな障害や葛藤が待ち受けることがある
- 物語のクライマックスに向けた緊張感の再構築

**10. 復活**
- 古い自分の死と新しい自分の誕生を象徴する瞬間
- 世界に対する新しい理解や価値観が示される

**11. 宝を持ち帰る**
- 宝は物理的なものだけでなく、知恵、成長、人間関係など多様
- 主人公の変化が周囲の世界や他者にも影響を与えることを示す

### 13.5 Eパート — 新しい日常

```
構成要素:
  ・新しい日常の世界
  ・主人公が得たもの
```

## 20. 既知の課題と制約事項

### 20.1 技術的課題

| # | 課題 | 詳細 |
|---|---|---|
| 1 | **レガシーAPI依存** | `openai==0.28`（ChatCompletion API）を使用しており、最新のOpenAI Python SDK（v1.x）への移行が必要 |
| 2 | **Cell 21の構文エラー** | `I visual_supporter = ...` に不要な `I` が存在する |
| 3 | **未定義関数の参照** | Cell 25-31で `api_inference()`, `save_to_sheet()`, `output_markdown()` が使用されているが定義が見当たらない |
| 4 | **変数名の不整合** | Cell 16で `{prot}` となるべき箇所が `{plot}` ではなく `{prot}` として参照されている |
| 5 | **`input`変数の競合** | Cell 8, 24で `input` をPython組み込み関数名と同名の変数として使用している（定義箇所不明） |
| 6 | **APIキーのハードコード** | APIキーが直接コード内に記載される構造であり、セキュリティリスクがある |

### 20.2 設計上の制約

| # | 制約 | 説明 |
|---|---|---|
| 1 | **Google Colab依存** | `google.colab.auth` に依存しており、ローカル環境では一部機能が動作しない |
| 2 | **APIコスト** | 100回ループでのキャラクター生成は1ループあたり最低5回のAPI呼び出し（合計500回以上）を伴い、高額になる可能性がある |
| 3 | **セッション間の状態保持なし** | Colab上の実行結果はセッション終了で消失する（Google Sheets保存は任意） |
| 4 | **エラーハンドリングの一律化** | すべてのAPI呼び出しで同一のtry-exceptパターンを使用しており、エラー種別に応じた処理分岐がない |
| 5 | **逐次実行モデル** | すべてのAPI呼び出しが同期的に逐次実行され、並列化されていない |

### 20.3 改善提案

1. OpenAI SDK v1.x への移行
2. 環境変数またはColab Secretsを用いたAPIキー管理
3. 未定義関数（`api_inference`, `save_to_sheet`, `output_markdown`）の実装または既存関数への統合
4. 構文エラーの修正（Cell 21）
5. 変数名の一貫性確保（`prot` → `plot`、`input` の改名）
6. 非同期APIコールによる並列化の検討
7. コスト見積もり機能の追加
